# Shinoda Agent Creation Plan (V1)

## Objective

Turn the skeletal `agents/shinoda` workspace into a functional Mastra-based AI orchestration agent that can observe, diagnose, and reason about the Pulse track-processing pipeline at runtime.

Shinoda is the platform's operational intelligence layer — it connects to the same NATS event plane and service APIs that the microservices use, and provides developer-facing tooling for pipeline visibility, event inspection, and track-level reasoning.

This plan is **implementation-ready but plan-only**. No Shinoda code is written in this phase.

---

## Scope

### In Scope

- Agent definition (system prompt, model, instructions)
- Tool implementations for pipeline observation and event inspection
- Knowledge base documents grounding the agent in Pulse domain context
- Workflow for multi-step pipeline debugging
- Mastra registration and dev-server configuration
- Acceptance criteria and validation commands per milestone

### Out of Scope

- Replacing shinod-ai microservice logic (Petrified/Fort Minor/Stereo remain as-is)
- Production deployment topology for the agent
- Frontend integration (Shinoda is developer/operator-facing, not end-user-facing)
- Mastra memory/RAG persistence configuration (deferred to V2)

## Architecture

### Agent Identity

| Property | Value |
| --- | --- |
| Name | `shinoda` |
| Framework | Mastra (`@mastra/core`) |
| Model | `openai/gpt-4o-mini` (cost-efficient for operational tooling; upgradeable) |
| Dev server | `mastra dev` on `http://localhost:4111` |
| Workspace | `agents/shinoda` |

### Relationship to Shinod AI Service

Shinoda (the agent) is **not** the same as shinod-ai (the microservice). The microservice runs Petrified/Fort Minor/Stereo as NestJS modules processing tracks through NATS events. The agent is a developer-facing Mastra application that observes and reasons about pipeline health. They share domain knowledge but have completely different runtime boundaries.

```
┌──────────────────────────────────────────────────────┐
│  Pulse Platform (runtime)                            │
│                                                      │
│  Soundgarden → Petrified → Fort Minor → Stereo       │
│       │            │            │           │        │
│       └────────────┴────────────┴───────────┘        │
│                    NATS event plane                   │
│                         │                            │
│  ┌──────────────────────┼──────────┐                 │
│  │  Backstage (observer) │          │                 │
│  │  - subscribes track.> │          │                 │
│  │  - projects pipeline  │          │                 │
│  │  - HTTP /pipelines    │          │                 │
│  └───────────────────────┘          │                 │
│                                     │                 │
│  ┌──────────────────────────────────┘                 │
│  │  Shinoda Agent (Mastra)                           │
│  │  - reads Backstage /pipelines API                 │
│  │  - inspects NATS event subjects                   │
│  │  - reasons about pipeline health                  │
│  │  - developer-facing via mastra dev                │
│  └───────────────────────────────────                │
└──────────────────────────────────────────────────────┘
```

### Integration Points

| Integration | Protocol | Purpose |
| --- | --- | --- |
| Backstage HTTP API | `GET /pipelines`, `GET /pipelines/:trackId` | Read pipeline projection state |
| Backstage Socket.IO | `pipeline.event` on `/pipeline` namespace | Live event stream for real-time diagnosis |
| Service health endpoints | `GET /health` per service | Container/service liveness checks |
| NATS (optional, future) | Direct JetStream consumer | Deep event inspection without Backstage proxy |

### Phase 3: Tools

**Goal:** Three operational tools that give Shinoda read access to live platform state.

#### Tool 1: `analyse-pipeline`

**Purpose:** Query Backstage HTTP API to retrieve pipeline state for a specific track or list recent pipelines.

**Input schema (Zod):**
- `trackId` (string, optional) — specific track to inspect
- `limit` (number, optional, default 10) — number of recent pipelines when no trackId

**Behavior:**
- `GET ${BACKSTAGE_URL}/pipelines/:trackId` when trackId provided
- `GET ${BACKSTAGE_URL}/pipelines?limit=N` for listing
- Return structured pipeline stage data (current stage, timestamps, event history)

**Error handling:**
- Return clear message if Backstage is unreachable
- Return "track not found" if no pipeline exists for the given trackId

**Environment:**
- `BACKSTAGE_URL` (default `http://localhost:4001`)

#### Tool 2: `inspect-events`

**Purpose:** Query recent events for a given NATS subject pattern or trackId by reading Backstage's event log.

**Input schema (Zod):**
- `subject` (string, optional) — NATS subject pattern (e.g. `track.petrified.*`)
- `trackId` (string, optional) — filter events by track
- `limit` (number, optional, default 20)

**Behavior:**
- Query Backstage pipeline events filtered by subject pattern or trackId
- Return chronologically ordered event list with timestamps and payload summaries
- Highlight gaps (e.g., fingerprint generated but no transcription started)

**Error handling:**
- Indicate when no events match the filter
- Suggest alternative subjects if a pattern yields no results

#### Tool 3: `check-services`

**Purpose:** Health-check all platform services and report their status.

**Input schema (Zod):**
- `services` (array of strings, optional) — specific services to check; defaults to all

**Behavior:**
- Hit `/health` or equivalent endpoint for each service in the topology
- Report status (healthy/unhealthy/unreachable) with response times
- Flag any service that is expected to be running but is not responding

**Service registry (hardcoded, configurable via env):**

| Service | Default URL |
| --- | --- |
| Authority | `http://localhost:7000` |
| Slim Shady | `http://localhost:7400` |
| Soundgarden | `http://localhost:7100` |
| Shinod AI | `http://localhost:7200` |
| Mockingbird | `http://localhost:7201` |
| Hybrid Storage | `http://localhost:7300` |
| Backstage | `http://localhost:4001` |

#### Acceptance Criteria

- Each tool has a Zod-validated input schema and clear output structure
- Tools handle network errors gracefully (no unhandled promise rejections)
- Tools are registered in the agent definition and visible in Mastra Studio
- `analyse-pipeline` returns accurate data when Backstage is running with pipeline data
- `check-services` correctly reports healthy/unhealthy for running/stopped services

### Phase 4: Debug Pipeline Workflow

**Goal:** A structured Mastra workflow that automates multi-step pipeline diagnosis.

#### Workflow: `debug-pipeline`

**Trigger:** Developer provides a `trackId` that is stuck or failed.

**Steps:**

1. **Gather context**
   - Call `analyse-pipeline` with the trackId
   - Call `check-services` for all services
   - Determine which pipeline stage the track is stuck at

2. **Identify gap**
   - Compare observed events against expected event sequence:
     `track.uploaded → track.petrified.generated → track.fort-minor.completed → track.approved → track.transcoding.completed → track.hls.generated → track.hls.stored`
   - Identify the last successful event and the expected next event
   - Check if the responsible service for the next event is healthy

3. **Diagnose**
   - If the responsible service is unhealthy: report service failure
   - If the service is healthy but no event was emitted: suggest investigating the service's consumer logs
   - If a failure event was emitted (e.g., `track.petrified.failed`): report the failure with any available error details
   - If duplicate detected: report `track.duplicate.detected` with context

4. **Report**
   - Return a structured diagnosis with:
     - Track ID and current pipeline state
     - Last successful stage and timestamp
     - Expected next stage and responsible service
     - Diagnosis summary and suggested action
     - Service health snapshot

#### Acceptance Criteria

- Workflow can be invoked from Mastra Studio with a trackId
- Correctly identifies stuck-at-stage for a track that stopped at `track.petrified.generated`
- Correctly identifies service health issues when a service is down
- Correctly interprets terminal failure states (`track.rejected`, `track.*.failed`, `track.duplicate.detected`)

#### Validation

```bash
# With infrastructure and at least one uploaded track:
cd agents/shinoda && pnpm dev
# In Studio: "Debug the pipeline for track abc-123"
```

## Technical Decisions

### Why Mastra

Mastra is already chosen and dependencies are installed. It provides:
- Agent definition with tool registration
- Workflow orchestration for multi-step operations
- Dev Studio for interactive testing (port 4111)
- Built-in observability hooks

### Why Backstage as Primary Data Source

Backstage already subscribes to `track.>` and projects all pipeline events into MongoDB. Using Backstage's HTTP API means:
- No additional NATS consumers needed
- Pipeline history is already aggregated and queryable
- No risk of competing with production consumers

Direct NATS access is deferred to a future phase if Backstage's API proves insufficient.

### Why GPT-4o-mini

Shinoda's tasks are primarily structured data analysis and pattern matching against known pipeline shapes. GPT-4o-mini is sufficient and cost-efficient. The model is configurable via environment variable if a stronger model is needed for complex reasoning.

## Future Considerations (V2+)

- **Memory persistence:** Use `@mastra/memory` with LibSQL to maintain conversation context across debugging sessions.
- **RAG over logs:** Index service logs for deeper root-cause analysis beyond event-level inspection.
- **Direct NATS consumer:** Subscribe to JetStream for real-time event streaming without Backstage intermediary.
- **Shinod AI microservice split companion:** When Petrified/Fort Minor/Stereo become independent services, update Shinoda's knowledge base and service registry accordingly.
- **Alerting workflow:** Automated pipeline-stall detection with notification (Slack/Discord webhook tool).
- **Evals:** Use `@mastra/evals` to score diagnostic accuracy against known pipeline scenarios.
