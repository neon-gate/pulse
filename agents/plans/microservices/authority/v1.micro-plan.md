# Authority Micro Plan (V1)

## Purpose

Realign `Authority` with the project pipeline purpose while preserving current stack and architecture:
- NestJS + Clean Architecture
- Ports and Adapters
- NATS event-driven integration

## Scope

- Keep ownership of auth identity, sessions, tokens, and OAuth.
- Keep `Authority` as source of authentication truth.
- Align contracts with `agents/context/EVENTS_PIPELINE.ms`.

## Current Responsibilities

- Signup/login/logout/refresh/me HTTP API
- Session persistence
- JWT issuance
- Emits:
  - `authority.user.signed_up`
  - `authority.user.logged_in`
  - `authority.token.refreshed`
  - `authority.user.logged_out`
- Consumes:
  - `user.profile.created`

## Contracts

### HTTP

- `POST /authority/signup`
- `POST /authority/login`
- `POST /authority/logout`
- `POST /authority/refresh-token`
- `GET /authority/me`

### Events

- Produces: `authority.user.signed_up`, `authority.user.logged_in`, `authority.token.refreshed`, `authority.user.logged_out`
- Consumes: `user.profile.created`

## Storage Ownership

- MongoDB: auth users, sessions, refresh/session state
- No ownership of profile aggregate persistence

## Alignment Gaps

- Ensure all outbound auth events include stable identity keys and timestamps.
- Ensure profile-linking from `user.profile.created` is idempotent and observable.
- Ensure JWT claims contract is consistent with downstream expectations.

## Plan

1. Validate and lock event payload schema for all authority outbound events.
2. Harden `user.profile.created` consumer idempotency and error handling.
3. Define canonical JWT claim policy for cross-service usage.
4. Add smoke-test scenarios for auth -> profile creation loop.

## Refactor Backlog

1. Add explicit zod/event-schema guards for all emitted authority payloads.
2. Make profile-linking consumer idempotency behavior explicit and observable.
3. Normalize token payload policy for `profileId` and future claims.

## Smoke-Test Checklist (Design Only)

- [ ] Signup emits `authority.user.signed_up` once per accepted request.
- [ ] Duplicate `user.profile.created` does not corrupt Authority user state.
- [ ] `/authority/me` returns stable contract including `profileId` after linkage.
- [ ] Refresh/logout lifecycle emits expected events and remains backward compatible.

## Acceptance Criteria

- Event contracts match `agents/context/EVENTS_PIPELINE.ms`.
- No auth responsibility leakage to profile service.
- Profile linkage survives duplicate events without corruption.
- Contract test checklist documented in final audit plan.
