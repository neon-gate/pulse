# Slim Shady Micro Plan (V1)

## Purpose

Align `Slim Shady` as the user-profile microservice, strictly separated from authentication concerns.

## Scope

- Keep ownership of user profile/public identity.
- Consume auth identity lifecycle events and maintain profile read/update APIs.
- Emit profile lifecycle events for dependent services.

## Current Responsibilities

- Profile create/read/update endpoints
- Profile preferences and onboarding state
- Consumes:
  - `authority.user.signed_up`
- Emits:
  - `user.profile.created`
  - `user.profile.updated`
  - `user.profile.deleted` (planned contract)

## Contracts

### HTTP

- `GET /users/profile`
- `PATCH /users/profile`
- `PATCH /users/profile/preferences`
- `POST /users/profile/onboarding/complete`

### Events

- Consumes: `authority.user.signed_up`
- Produces: `user.profile.created`, `user.profile.updated`, `user.profile.deleted` (planned emission)

## Storage Ownership

- MongoDB: profile aggregate, preferences, onboarding state
- No ownership of auth credentials/sessions

## Alignment Gaps

- Keep event payloads canonical and idempotent.
- Ensure profile identifiers are consistently mapped from auth identity.
- Keep no password/JWT/session logic inside this service.

## Plan

1. Confirm canonical profile contract in `agents/context/EVENTS_PIPELINE.ms`.
2. Enforce idempotent profile creation from signup events.
3. Normalize update event payload (`fields`, timestamp, profileId).
4. Add profile contract checks for Authority integration.

## Refactor Backlog

1. Stabilize profile event payload schema (`profileId`, `authUserId`, timestamps, changed fields).
2. Make signup-driven profile creation retries/duplicates idempotent and traceable.
3. Decide whether `user.profile.deleted` remains planned or is implemented in runtime.

## Smoke-Test Checklist (Design Only)

- [ ] `authority.user.signed_up` creates a single profile idempotently.
- [ ] Profile update endpoint emits `user.profile.updated` with deterministic payload.
- [ ] Authority receives `user.profile.created` and stores `profileId`.
- [ ] Slim Shady service has no auth/session writes.

## Acceptance Criteria

- Profile lifecycle contracts are stable and documented.
- `authority.user.signed_up` -> `user.profile.created` loop is reliable.
- Service boundary with Authority remains clean (no auth concerns in Slim Shady).
