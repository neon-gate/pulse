# Shinod AI Split Micro Plan (V1)

## Purpose

Plan the split of current `shinod-ai` modular service into **three independent microservices** while preserving existing architecture principles.

Target microservices:
- `petrified`
- `fort-minor`
- `stereo`

## Mandatory Constraints

- No infra sharing between the three microservices.
- Independent deployment/runtime boundaries.
- Communication only through EDA over NATS subjects.
- Follow microservice Clean Architecture pattern (`application`, `domain`, `infra`, `interface`).

## Current Responsibilities (Monolith-in-service)

- Petrified: fingerprint + audio hash + duplicate detection
- Fort Minor: transcription pipeline
- Stereo: decision reasoning and approval/rejection

## Contracts

### HTTP

- `GET /health` (operational endpoint)

### Events

- Petrified consumes `track.uploaded`, emits `track.petrified.*` and duplicate/failure events
- Fort Minor consumes `track.petrified.generated`, emits `track.fort-minor.*`
- Stereo consumes `track.petrified.generated` and `track.fort-minor.completed`, emits `track.approved|track.rejected|track.stereo.failed`

## Storage Ownership

- Current shared service ownership: MongoDB + Redis + object storage refs
- Split target: each of Petrified/Fort Minor/Stereo owns independent infra and persistence concerns

## Alignment Gaps

- Current fan-out and queue-group behavior can break stage delivery.
- Contracts between stages are implicit in shared runtime assumptions.
- Artifact references required downstream are not always propagated.

## Split Plan

1. Define service contracts per stage in `agents/context/EVENTS_PIPELINE.ms`.
2. Extract Petrified as independent service:
   - owns fingerprint generation and duplicate events
3. Extract Fort Minor as independent service:
   - owns transcription and completion/failure events
4. Extract Stereo as independent service:
   - owns readiness aggregation and approval/rejection decisions
5. Define migration sequencing and dual-run compatibility strategy.

## Refactor Backlog

1. Remove queue-group competing-consumer hazard on `track.petrified.generated`.
2. Stabilize handoff payloads so each stage has explicit required fields.
3. Split deployable units with independent env/config/infra adapters.
4. Define compatibility window and rollback strategy during split migration.

## Smoke-Test Checklist (Design Only)

- [ ] Every `track.uploaded` reaches Petrified, Fort Minor, and Stereo readiness path.
- [ ] `track.approved` carries enough source artifact metadata for Mockingbird.
- [ ] Reject/duplicate paths are terminal and observable in Backstage.
- [ ] Split services run independently with NATS-only communication contracts.

## Acceptance Criteria

- Each stage runs as an independent microservice with isolated infra.
- Stage-to-stage dependencies are only event contracts over NATS.
- No direct infra imports/shared adapters between the three services.
- End-to-end AI decision flow is explicitly mapped and testable.
